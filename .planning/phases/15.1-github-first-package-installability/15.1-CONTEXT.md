# Phase 15.1: GitHub-first package installability

## Purpose

Make every package boundary GitHub-installable by removing monorepo-only assumptions before the repo split in Phase 16. This phase is about install mechanics, package metadata, and verification. It is not a product-scope expansion.

## Locked decisions

- GitHub is the canonical distribution channel for package consumption. npm publishing is not the target path.
- Submodules and dedicated GitHub repos are acceptable install surfaces when a package or app is too heavy for direct package-manager installation.
- `workspace:*` is treated as a blocker because it ties installation to pnpm workspace semantics.
- We should prefer package metadata that works both inside this monorepo and in a future split-repo/submodule topology.
- We want explicit diagnostics for GitHub install failures so we stop repeating the same packaging mistakes.

## Primary risks

- Replacing `workspace:*` with `file:` dependency specs removes a pnpm-specific restriction and preserves local repo installs, but it does not by itself solve cross-repo GitHub dependency transport for every internal package.
- Packages that expose `dist` artifacts but do not build on install will appear installable locally and fail when fetched from GitHub.
- Source-distributed TypeScript packages need explicit exports and file lists or subpath imports will break outside the workspace.
- Apps should be treated as repo/submodule install surfaces, not forced into a registry-style package model.

## Execution order

1. Harden reusable package manifests and add install diagnostics.
2. Verify package installs from GitHub-style conditions.
3. Close remaining app/outlier blockers with repo/submodule decisions before Phase 16 proceeds.
